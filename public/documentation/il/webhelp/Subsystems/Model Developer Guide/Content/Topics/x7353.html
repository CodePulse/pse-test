Configuring mathematical formulation and validation

Before executing a model-based activity, gPROMS can perform various checks and modifications to the mathematical formulation of the problem. These are summarised below.

IdentityElimination: ON (default) or OFF
If set to ON, the solver will attempt to reduce the size of the problem internally by removing equations of the form , and substituting all occurrences of one of these variables with the other. Such equations are often introduced in gPROMS models by stream connectivity equations. This may result in a faster solution although at the current stage of development the costs of creating and using the reduced system sometimes outweigh the benefits, particularly when many IF and CASE conditions are present.
IgnoreDAEIndexAnalysis: ON or OFF (default)
Determines whether or not gPROMS attempts to (re)initialise a system if the index is determined to be greater than 1.
IndexReduction: ON or OFF (default)
Specifies whether or not gPROMS will perform automatic index reduction for high-index models. This Solution Parameter applies to all four activities. See High-Index DAE systems for more details.
PerformStrictDAEStructuralChecks: ON or OFF (default)

If set to ON, gPROMS performs a check of the structure of the DAE system following the first initialisation and following each reinitialisation.

This option is useful because unchecked structural errors can lead to different symptoms (such as an error reported by the Linear Algebra solver) which can be hard to diagnose. It can be switched on during Model development in order to detect Modelling errors before they result in (harder to diagnose) numerical errors. It is particularly useful for Models containing IF or CASE statements.

Once the Model has been well tested, this option can be turned off, as it may have some computational overhead for complex models, particularly those that contain IF or CASE statements that change branch frequently during simulation. When a (re)initialisation fails a strict DAE structural check will be performed even if PerformStrictDAEStructuralChecks := OFF.

ModelPruning: ON (default) or OFF

Determines if gPROMS automatically reduces the model to the minimum required to produce the information requested by the gPROMS Activity specification. As a result of the reduction, the smallest possible mathematical problem is posed to the numerical solvers.

Using this option usually results in a shorter computation time, although the effort of reducing the model may outweigh the gains, particularly for short horizons in dynamic models.

As a common benefit across all Activities, general purpose models do not need to be pruned manually depending on the Activity for maximum performance gain; this is automatically taken care of.

The table below lists the effects of model pruning for each gPROMS Activity.

gPROMS Activity Effects of model pruning
Simulation

Variables that can be pre-calculated are computed only once at the beginning of execution.

Variables that can be post-calculated are taken into account only at the end of successful time-steps.

Optimisation

Variables or decision variables that affect neither the objective function nor constraints are automatically eliminated.

Parameter Estimation

Variables or estimated parameters that do not affect the measured variables are automatically eliminated.

Design of experiments

Variables or experimental controls that do not affect the measured variables are automatically eliminated.

All Equations are not generated for which all variables have been eliminated.
