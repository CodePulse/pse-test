Installing Foreign Processes

Once the Foreign Process code has been written and compiled as explained in the previous sections, the resulting dynamic library has to be installed in a place where gPROMS can find it during runtime.

There are three main options for this:

  • In a sub-directory called "fpi" of the gPROMS installation directory.

  • In a gPROMS project file.

  • In a sub-directory called "fpi" of the user's current gPROMS working directory. This sub-directory is at the same level as the "input", "output" and "save" sub-directories.

The first option makes the Foreign Process Interface accessible to all gPROMS users but will usually require system administrator privileges. The second option allows to easily store and disseminate a Foreign Process Interface together with the project where it is used. However, this has a couple of disadvantages in that it must be stored in every project that is supposed to use it. Secondly, on Windows this may cause conflicts because of an incorrect platform (32 vs. 64-bit).

The last option is mentioned only for completeness sake because it is appropriate only in very rare cases, for instance when deploying gPROMS-based applications or when gO:RUN is used and the gPROMS installation cannot be modified.
