Creating a custom databank for SAFT-VR SW

In the case of a databank that corresponds to the SAFT-VR SW EoS, the <gsaft> element is required to have a single <compounds> element as a child element. The <compounds> element is only allowed to contain a single or multiple <compound> elements, with each element containing the parameters that are required to describe a compound.

The <gsaft> element can further contain a single <crossInteractions> element. The <crossInteractions> element can contain multiple <crossInteraction> elements that are used to specify the interaction parameters between different compounds. Note that the <crossInteraction> element is always in reference to declared compounds in the <compounds> element, and the specification of the compounds to which a <crossInteraction> element refers to is done in terms of the compound1 and compound2 attributes. A small example that demonstrates how a databank that specifies the compounds water, n-decane, and methanol as well as the cross interactions between these is shown below. Note that the contents of the <compound> and <crossInteraction> elements are ommited for brevity; these are discussed in more detail in the following sections. 

<gsaft theory="vr_sw">
  <compounds>
    <compound name="water">
      ...
    </compound>
    <compound name="n-decane">
      ...
    </compound>
    <compound name="methanol">
      ...
    </compound>
  </compounds>
  <crossInteractions>
    <crossInteraction compound1="water" compound2="n-decane">
      ...
    </crossInteraction>
    <crossInteraction compound1="water" compound2="methanol">
      ...
    </crossInteraction>
  </crossInteractions>
</gsaft>
Compound definition

In a databank that is to be used with the SAFT-VR SW EoS each compound is specified within a <compound> element. A valid <compound> element requires the specification of the name of the compound by setting the name attribute to a valid string. Each compound must have a unique name. Each <compound> element can contain any of the elements presented in the table below, where each element can be of single occurrence. Note that, as listed in the Element Type column, elements can be of two types. Simple elements are elements that simply contain a value (e.g., <mw>18.015</mw>) and have no attributes and/or child elements. Any other element is listed as Complex; the exact structure of these elements will be discussed in more detail.

The elements of complex type that a <compound> element can contain are:

The <dielectric> element

This element gives the coefficients of a linear correlation for the value of the dielectric constant as a function of temperature. The parameters v and t are specified as the values of the corresponding attributes of the <dielectric> element, as shown below for the case of water as an example.

<gsaft theory = "vr_sw">
  <compounds>							
    <compound name = "water" >
      <dielectric v = "0.3777e-3" t = "1403" />										  
      ...
    </compound>
  </compounds>
</gsaft>

The <heatCapacity> element

This element contains the values of the coefficients for the polynomial expression used to calculate the ideal gas heat capacity of a compound as a function of temperature. The values of four coefficients need to be provided (a, b, c, d) as separate elements within the <heatCapacity> element, as shown below for the case of water.

<gsaft theory = "vr_sw">
  <compounds>
    <compound name = "water" >
      ...											
      <heatCapacity>
        <a>32.2455</a>
        <b>1.923835e-03</b>											
        <c>1.055492e-05</c>																						
        <d>-3.596461e-09</d>																																	
      </heatCapacity>
      ...
    </compound>
  </compounds>
</gsaft>

The <associationSites> element

This element is optional and is used in the case of associating (hydrogen-bonding) compounds for defining the different types of association sites on a compound, as well as the multiplicity of each association site type. The element can contain multiple <siteMultiplicity> elements, with each element used for the specification of each site type. The site type is specified as the value of the name attribute of each <siteMultiplicity> element and is required to be a valid string. The multiplicity of each association site type is then specified by setting the value of the <siteMultiplicity> element accordingly, which is required to be a positive integer. Looking at the example of water, the association scheme speficied is summarised in the table below.

The <selfInteraction> element

The <selfInteraction> element contains the parameters that describe the self interactions between a given compound. These are grouped in two different elements, with the <dispersion> element containing the parameters that describe the dispersion (in this case square-well) interactions, and the <association> element, where the parameters that describe the association interactions are placed. Note that the <dispersion> element is required within a <selfInteraction> element, whereas the <association> element is optional and can only be declared for associating compounds.

The <dispersion> element contains the parameters of the square-well interaction potential for the self-interaction of the compound. Together with the <numberOfSpheres> element, the <dispersion> element provides the set of parameters that are required by SAFT-VR SW. The parameters that are specified within the <dispersion> element are listed in the table below.

The <association> element is used for the specification of the parameters that describe the association interactions (modelled by the square-well potential) between site types of a compound. Each site-site interaction is described by an <interaction> element, whereby the site types to which the interaction refers are declared as the values of the site1 and site2 attributes of the <interaction> element. Note that the specified site types must have been previously declared in the <associationSites> element of the compound, as discussed above.

The association interaction requires the specification of two parameters per interaction, which are summarised in the table below.

Notes:

  • In the example presented above the <interaction> element describes the association interaction between sites of different type. Association interactions between sites of the same type can also be declared in the same fashion, simply by assigning the same value to both the site1 and site2 attributes of the <interaction> element.
  • Site types can be assigned to the site1 and site2 attributes of the <interaction> element in any order; the association interactions for site types of the same compound are symmetric, i.e., either of the elements <interaction site1="O" site2="H"> and <interaction site1="H" site2="O"> can be used to specify the same interaction.
Cross-interaction specification

The specification of cross interactions between compounds is optional, and each <crossInteraction> element is used to specify the interaction, both for dispersion and association interactions, between two compounds. The compounds to which each <crossInteraction> element refers to are specified by assigning the corresponding compound names to the compound1 and compound2 attributes of the <crossInteraction> element. Note that:

  • The values (i.e. compound names) assigned to the compound1 and compound2 attributes must be valid strings and must correspond to names of compounds previously declared in the databank within the <compounds> element
  • A cross interaction between two compounds can be specified by assigning the corresponding compound names to the compound1 and compound2 attributes in any order. However, each cross interaction must have a unique definition, i.e., only one <crossInteraction> element that refers to two specific compounds is allowed in the databank.

The structure of a <crossInteraction> element follows closely the case of the <selfInteraction> element.

The specification of the cross dispersion interactions is done within a <dispersion> element. As for the case of self interactions, the <dispersion> element can contain single <epsilon>, <sigma>, or <lambda> elements. The specification of the parameters is optional, and if they are not specified the corresponding parameter values are calculated by means of the appropriate combining rules. 1 For more details on the combining rules employed see: A. Galindo, L. A. Davies, A. Gil-Villegas, and G. Jackson, “The thermodynamics of mixtures and the corresponding mixing rules in the SAFT-VR approach for potentials of variable range”, Molecular Physics, 93, 241-252 (1998)

The specification of cross association interactions is done within the <association> element, where a single <interaction> element is needed for the interaction between two site types. For the specification of the association interaction between site types on different compounds the order in which the site types are declared in the site1 and site2 attributes of the <interaction> element needs to follow the specification of the compounds (compound1 and compound2 attributes) in the <crossInteraction> element. Namely, the site type specified in the site1 attribute, must be a site type declared within the compound referenced in the compound1 attribute of the <crossInteraction> element. The same holds for the site2 and compound2 attributes. Note that within each <interaction> element the specification of both parameters (epsilonAssoc, and rc) within the corresponding elements is required; no combining rules apply in this case and any parameter not declared is assigned the value 0.
