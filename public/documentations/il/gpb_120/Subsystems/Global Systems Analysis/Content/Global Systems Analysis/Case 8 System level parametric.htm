Case 8: System level parametric case study 

A system level parametric analysis can be performed to gain insight and provide support for system level decisions. For example, it is possible to perform a system level parametric analysis to determine how coolant water affects the total cost of a system, or how equipment sizes affect profit. Usually, multiple simulations with different parameter values sets are required but GSA makes it possible to perform a parametric analysis at system level with a single GSA execution.

The purpose of this example is to investigate the effects of cross section area, pipe height and inlet molar flow rate on yield and temperature.

Create a GSA entity named 'Case8_SystemParametric'. The settings for the General tab are shown in the figure below. In this example, uncertainty analysis is performed with the quasi-random Sobol sample generation method.

Case 8: General tab

Specify the system level parameters of interest as factors in the Factors tab. Either use the 'Select...' button or enter the full variable path into the 'Variable in model' column, pressing Control+Space to use autocompletion. Note that, since only variables are eligible as factors, all parameters of interest must be moved from the PARAMETER section to the VARIABLE section of the model before the GSA entity is set up.

Select each factor and click on the 'Edit...' button to specify all other factor settings. Note that, in a 'Gridded set' distribution, the number of intervals does not equal the total number of sample points of the factor (number of sample points = number of intervals + 1).

In practice, since the size of a CSTR is limited to a particular set of options due to manufacturing constraints, it is more appropriate to specify the distribution of factor 'cross_section_area' as 'Enumerated set' in reality. However, 'Gridded set' is specified in this example for the sake of simplicity.

As there are no sampling restrictions in this example, leave the 'Sampling restrictions' section blank.

Specify the number of uncertainty scenarios in the 'Number of samples' section. In the absence of uncertainty scenarios, leave it at the default value of one. The total number of realisations of the base activity to be calculated is given by:

  Nd Number of deterministic cases
  N Number of uncertainty scenarios (user defined) 

Specify the responses in the Responses tab. Similar to factors, either use the 'Select...' button or enter the full variable path directly into the 'Variable in model' column, pressing Control+Space to use autocompletion.

Case 8: Responses tab

The 'GSA language' tab is automatically updated once the General, Factors and Responses tabs are set up. Launch the execution by right-clicking on the GSA entity and selecting 'Analyse system' or pressing icon in the toolbar. 

Case 8: Executing GSA

The 'Analyse System' dialog box will appear. Change the Case name and select the desired number of workers for parallel computation. Note that this number should not exceed the total number of logical cores on the machine.

Once execution completes successfully, right-click on the 'System Analysis Group' folder and select 'New result element...' to create result elements. Alternatively, copy and paste result elements from other Cases. Note that the 'Factor Sensitivity Table' is not applicable in the context of uncertainty analysis and will not be populated with results.

Case 8: Creating a result element

The 'Scatter 2d' result element can be used to illustrate the relationship between any two factors, responses or a factor and a response. Select which variables to show on the X and Y axes via the Attributes, shown in the figure below.
