Running the Parameter Estimation activity

You can execute a Parameter Estimation activity in a number of ways. First, select the Estimation entity from the project tree, and then either;

  1. drop down the Activities menu and select Estimate... , or

  2. click in the tool bar, or

  3. press F5.

If there are any syntactical or cross-referencing mistakes etc., these will be reported. Otherwise, the execution options dialog (below) will be displayed. Upon clicking OK, an execution Case is created and the activity begins.

The Parameter Estimation activity execution control dialog

Parameter Estimation reporting intervals: Most of the execution options for Parameter Estimation are identical to those for Simulation. The only extra option is in the reporting interval. In addition to the reporting interval period, you may also select at which stages of the optimisation the values are recorded into the results trajectory. The three options are:

  • Final results - no values are recorded until an optimal solution has been found.

  • Major iterations - values are recorded at every major iteration; that is, after a set of parameter values is accepted and a new search direction is being computed.

  • Every step - as above, and in addition, values are recorded during the line-search steps of the optimisation. This may include experiment evaluations that do not complete - and so may be useful for diagnosing problems, such as problematic combinations of parameters.

Note

In addition to the reporting interval, the values of monitored variables are also recorded at each measurement time of the Performed Experiment. Therefore, if you wish to view the values of monitored variables at just the measurement times, simply enter a large reporting interval.

See also the Variable Trajectories section.

Number of workers for parallel computation: This specifies how many experiments are evaluated in parallel at each line search step. The maximum is 32, the minimum is 1. If a number larger than 1 is specified, a so-called HPC (High Performance Computing) license is requested.

Some considerations regarding the choice of workers:

  • The number should exceed neither the number of logical cores nor the number of experiments.

  • The workers running in parallel should not consume more than the available physical memory (RAM) minus a generous safety margin for the operating system and other applications that are active as well. Otherwise the execution will effectively slow down because the operating system needs to swap fast RAM with slow disk memory.

    Each operating system provides tools for assessing the available RAM at execution time, on Windows this is, for instance, possible with the Windows Task Manager.
