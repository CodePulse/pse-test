Executing and resuming a GSA activity

Before the execution of a GSA activity can be launched, a number of prerequisites must be satisfied:

  • The GSA entity editor refers to a valid base activity (currently limited to Processes).
  • All factors and responses are valid for the specified base activity.
  • At least one factor is enabled.
  • At least one response is enabled.
  • A non-empty sample set is defined (that is, the combination of all enabled sampling restrictions must define a non-empty and feasible set).

The entity editor will display warning messages whenever one or more prerequisites are not satisfied.

Note that the Schedule section of the Process will also be executed. It is therefore advisable to remove the Schedule section for steady-state analysis. Alternatively, it is possible to choose to keep the Schedule and use GSA to investigate the effect of factors on the responses at the end of a scenario, for example, at the end of a batch recipe. See a workaround to obtain intermediate results.

When all prerequisites have been met, it is possible to launch an execution by selecting the GSA entity and clicking on the 'Analyse System' button on the toolbar, via the Activities or the entity context menu, or by pressing F5. It is then possible to configure the hardware resources (currently the number of logical processors) to use in the analysis.

Selecting the number of workers for parallel computation in GSA

The number of workers for parallel computation should not exceed the number of logical processors available. Due to hyper- or multi-threading, the computer device being used may have a number of logical processors, which is larger than the number of physical cores. Note that using more than one worker requires an HPC license.

Based on the specification, a GSA manager will spawn GSA workers for each logical processor, which will compute the realisations of the base activity in parallel. GSA currently supports multiple processors on a single core.

As the factor sample space is covered and realisations completed, the completion percentage will be reported in the execution output, with a time stamp and at regular intervals. At each realisation, the values of factors and responses are recorded in HDF5 format, suitable for very large data sets. Once execution has completed, the HDF5 file is displayed in the Results folder within the Case, in addition to a report containing useful information. Note that the GSA execution is resilient to point failures in simulations, and the sampler will attempt to continue sampling until the requested number of successful samples has been achieved. The results can be explored by configuring result elements in the 'System Analysis Results' folder.

GSA execution can be stopped at any time by clicking on the toolbar button typically used to abort execution, or by right-clicking on the Case and selecting 'Stop execution...'. Once stopped, execution can be resumed by right-clicking on the Case and selecting 'Resume analysis'.
