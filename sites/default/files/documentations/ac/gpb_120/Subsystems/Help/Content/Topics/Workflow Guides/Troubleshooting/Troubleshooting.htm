Troubleshooting

Overview

Troubleshooting is an integral part of constructing complex models. Errors made during the construction of the flowsheet model typically manifest themselves when an attempt is made to simulate the flowsheet model. It may fail to simulate for one or more of the following reasons:

  • it is an incorrect model of the physical process that has no solution
  • there is no physical solution to the problem or specification trade-offs have been made that are not feasible
  • the solution does exist but the flowsheet model has been incorrectly set up
  • the solution does exist but gPROMS ProcessBuilder is unable to produce this solution

For more complex models it can be difficult to distinguish between these causes of failure or to find in which part of the flowsheet model an error has been made. Mistakes can be made during the construction of the flowsheet model in the values that are entered and the selections made in dialogs of unit models. These may range from specifying certain quantities in the wrong units of measurement, imposing conditions on the model outputs that are infeasible to omitting required specifications.

Even if the model is formulated correctly, it may be set up in such a way that the solution algorithm will have difficulty finding a solution. Examples of this are, failing to insert a recycle breaker unit (see 'Flowsheets with recycles') in a recycle stream or providing incorrect initial guess values for recycle streams.

Even if the flowsheet model has been set up correctly, gPROMS ProcessBuilder may be unable to solve the problem. This might be the case for complex flowsheets or when physical properties packages used in the flowsheet model are not able to produce properties for intermediate solutions (of the flowsheet model solution algorithm).

The gPROMS language (see 'Custom modelling in gPROMS ProcessBuilder') can be used to modify correlations in gML models, to construct custom unit operation models or to formulate overall process KPI's. Incorrect use of the gPROMS language may also be a source of errors.

In many instances, information provided in the "Execution output" window can be used to analyse a failing simulation. This tutorial provides an overview of the steps to follow in order to analyse information in the "Execution output" window and to resolve errors.

Hint

Because debugging of complex models of chemical processes can be difficult, it is often good to construct the flowsheet model incrementally and simulate it at every step to verify that the model can be solved and that realistic output is produced. Each incremental step can be saved to disk as a separate project or, when using a version control system, as a separate revision.

When this approach is adopted, and a failure does occur, an efficient strategy to find the problem is to compare the current version of the model with a previous version that was working as intended. The built-in compare tool can be used to compare flowsheet models and processes. The compare tool can be accessed using Tools -> Compare. When specifications, in dialogs, have been changed it is good practice to compare the process entities for both versions of a project.

Problem panel

Before attempting to run a simulation, make sure that there are no errors or warnings displayed on the problem panel that appears during flowsheet construction. These errors or warnings will generally result in failure during the simulation. To check the problem panel, double-click on the flowsheet model in the project tree. If there are any errors or warnings, these will be shown in the bottom panel. To correct errors and warnings, click on the error or warning in the problem panel. The topology window will be centred on the unit model for which the warning or error is flagged up. Change the port connections or information entered on the dialog for the unit model to correct the error.

Warnings shown in the problem panel when unit model, within the flowsheet, have not been correctly connected

Failed simulations

After starting the simulation there may be a delay (depending on the complexity of the process) before the simulation is completed. The simulation may either complete successfully or fail. When a simulation has failed a notification pop-up dialog will be shown.

Example of a simulation failure dialog

In addition, the case project will contain a red cross. When the simulation has failed either no results are generated or partial results are generated depending on which type of failure has occurred.

Example of a Case file for a failed simulation

Finally, the execution output will indicate that the simulation has failed. The failure message shown depends on which type of failure has occurred. The key to resolving problems that occur during the model execution stage is to understand common types of errors and how to distinguish and troubleshoot these.

Example of the execution output window, indicating an error has occurred in the simulation

Analysing the execution output

The execution output window shows information related to the solution of the flowsheet model. This solution is performed in a number of stages that are executed in sequence. If all the stages are completed successfully, a solution is obtained.

For a steady-state flowsheet, the process will contain the following stages:

  • Correctness check of the model formulation
  • Simulation license check
  • Resolution of the model parameters
  • Initialisation of foreign objects
  • Degree of freedom analysis
  • Convergence of the unit model equations at the start of the flowsheet initialisation procedure
  • Flowsheet initialisation procedure to obtain the desired solution

For dynamic simulation there are two additional stages:

  • Time integration of the flowsheet model equations
  • Re-convergence of the flowsheet model equation upon encountering a discontinuity

To solve a problem that leads to a simulation failure, first determine during which stage of the solution process the problem occurs. In the execution output window, the error message is, typically, the last message before the simulation ends. Look for the execution messages, at the start of each stage, and determine whether or not they are followed by an error message before the start of the next stage.

Hint

It is possible to search for the text of an error message in the execution output. To do this, select Tools -> Search and replace or press F11. A window will appear at the bottom of the screen and a search string can be entered. After pressing 'Search' all the instances of the string in the execution output will be shown. Clicking on an instance in the window will center the execution output on that instance.

Correctness check of the model formulation

During this stage, checks are performed that verify that every variable, that is referred to, actually exists. In addition it is checked that there are no errors made in the gPROMS language or the simulation schedule. Error messages, displayed for problems during this stage, often explicitly state the nature of the problem. These problems are therefore generally simple to resolve.

Simulation license check

Running a simulation generally requires a license. Prior to execution a simulation, a check is made that this license is available. The execution of a simulation, of a flowsheet model involving gML unit models, generally requires the following licenses:

  • general simulation activity license
  • licenses for the execution of gML library models
  • licenses for any physical property package defined in the flowsheet model

When the first two types of licenses are not available, this will be indicated directly when the simulation starts. The availability of the latter type of license will only be checked during the "Initialisation of foreign objects" stage.

Dialog that is shown when a license for the execution of gML library models is not available.

The type of license that is not available will be indicated in the execution output window:

Acquiring licenses for protected models ...
Requesting 1145_PB_SEP_ADSORPTION_X license from server.
No license currently available.

Error message displayed in the execution output indicating which types of license is not available.

To troubleshoot licensing problems, verify the following:

  • Are appropriate licenses used by other users within the organisation
  • Are appropriate licenses in use by other simulations that have not terminated yet or which have terminated but are holding their license for diagnostic purposes
  • Are appropriate types of licenses available for the simulation activity that is started
  • Are appropriate licenses still valid (i.e. not expired)

For assistance with licensing problems refer to PSE support.

Resolution of the model parameters

Prior to constructing a system of equations, all the parameters in the model are calculated. Parameters have a fixed value that does not change during a simulation. For problems during this stage, see 'Resolution of the model parameters'

Initialisation of foreign objects

A flowsheet model simulation uses unit models that are primarily written as equations in the gPROMS modelling language. However, flowsheet model simulations generally also rely on calculations performed by compiled code, whether this code is a part of gPROMS ProcessBuilder or provided by an external program. These pieces of code are known to gPROMS ProcessBuilder as foreign objects. For example, a physical property package is accessed as a foreign object.

When a simulation is started these programs are instantiated. This instantiation can fail for different reasons. It may be that definition string, for that foreign object, is incorrect either because it points to a foreign object file that does not exist or options, passed for that foreign object, are incorrect. Additionally, a program accessed as a foreign object may require its own license. If this license is unavailable the foreign object will not initialise. Each external program may generate its own error messages. Study these messages to resolve the problem with the instantiation.

Constructing the system

A flowsheet model consists of a number of mathematical equations which are solved together for a number of variables. At this stage this system is constructed.

A problem that can occur during this stage is that some variables that are arrays are indexed out of their bounds. This typically arises because invalid selections have been made in unit models in which variables are selected from "Bus" connections. A possible cause is that, at first, a unit model that selects a variable from a "Bus" connection is connected to a certain unit operation model which includes this variable. Next, the unit model is connected to a different unit operation model for which the selected variable is invalid, but the selection is not updated.

This type of problem can be difficult to correct on large flowsheet as no indication is given to which unit model on the flowsheet the error pertains and no warning is given in the problem panel. To identify the unit model causing the error, check unit models in which selections are made from "Bus" connections, for orange borders around selections. These indicate that the selection is not current for that model. To correct this, update the selection to select a variable from the dropdown list.

Degree of freedom analysis

After the system of equations has been constructed, some checks are performed to determine whether or not the system of equations is valid. This referred to as a "degree of freedom analysis". There are a number of problems that may results in a system of equations not being valid. Typically these arise from either making too many specifications for variables, or not enough specifications. See 'Degrees of freedom problems' for a detailed overview of problems that can occur during this stage and how to resolve them.

Solving the unit models

When the system of equations for the flowsheet model is valid, an attempt can be made to solve this system of equations. For steady-state, flow-driven, problems the first stage in solving these equations is to solve the equations for each unit model in the flowsheet. The unit models are solved in sequence. See 'Solving the unit models' for a detailed overview of problems that can occur during this stage and how to resolve them.

Solving the flowsheet

After a solution has been found for individual unit models, a solution will be sought for the complete flowsheet including recycles and specification trade-offs. See 'Solving the flowsheet' for a detailed overview regarding which problems can occur during this stage and how to solve these.
